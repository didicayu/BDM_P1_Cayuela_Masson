\section{Context and Objective}
This document summarizes the implementation delivered for P1.2 (Basic Ingestion and Landing) of the CyberSecIntel project.
It is aligned with:
\begin{itemize}[leftmargin=1.5em]
  \item \texttt{Project Statement - P1.pdf}
  \item \texttt{Project Statement - Context.pdf}
  \item previous team submission \texttt{BDM\_Project\_P1\_1.pdf}
\end{itemize}

Main objective for this increment:
\begin{itemize}[leftmargin=1.5em]
  \item automate ingestion from multiple cybersecurity sources,
  \item land raw data with traceable metadata in object storage,
  \item show clear API-vs-download source feasibility,
  \item prepare a scalable base for trusted/exploitation layers in P2.
\end{itemize}

\section{How Ingestion Into Landing Is Organized}
The ingestion layer is organized around two Airflow DAGs with clear responsibilities:
\begin{itemize}[leftmargin=1.5em]
  \item \textbf{\texttt{cybersecintel\_p1\_ingestion\_landing}}: dataset-import DAG for files staged in \texttt{data/raw\_downloads} (CIC CSV and CIC PCAP).
  \item \textbf{\texttt{cybersecintel\_api\_expansion\_ingestion}}: non-local-source DAG for API/feeds (KEV, NVD, URLhaus, EPSS, CIRCL, ThreatFox, seeded Shodan) plus stream placeholder ingestion.
\end{itemize}

Landing writes are implemented through a storage abstraction with backend set to MinIO (\texttt{LANDING\_BACKEND=minio}).
Data is stored directly in \texttt{s3://landing} (no local landing duplication).

Operational rules currently implemented:
\begin{itemize}[leftmargin=1.5em]
  \item partitioning by \texttt{ingest\_date=YYYY-MM-DD} (and \texttt{hour=HH} for stream),
  \item stable object naming per source/partition,
  \item prefix cleanup before each source write on same partition to avoid accumulation of duplicate files,
  \item sidecar metadata and per-source manifest update for traceability.
\end{itemize}

\section{Landing Zone Structure and Metadata}
Main prefixes currently in use:
\begin{itemize}[leftmargin=1.5em]
  \item \texttt{structured/kev/}, \texttt{structured/epss/}, \texttt{structured/cic\_ids2017/}
  \item \texttt{semi\_structured/nvd/}, \texttt{semi\_structured/urlhaus/}, \texttt{semi\_structured/circl\_vulnlookup/}, \texttt{semi\_structured/threatfox/}, \texttt{semi\_structured/shodan\_seeded/}
  \item \texttt{unstructured/pcap/source=cic\_ids2017/}
  \item \texttt{stream/ids\_alerts/}
  \item \texttt{metadata/manifests/}
\end{itemize}

Each ingested artifact is accompanied by:
\begin{itemize}[leftmargin=1.5em]
  \item raw source-native payload,
  \item \texttt{*.meta.json} with source URL, retrieval timestamp, checksum, size, and source-specific counters,
  \item manifest entry (\texttt{metadata/manifests/ingest\_date=.../<source>.jsonl}) for auditability.
\end{itemize}

\section{Hot, Warm, and Cold Paths}
\textbf{Cold path (implemented):}
\begin{itemize}[leftmargin=1.5em]
  \item persistent raw and semi-processed source data in MinIO landing,
  \item partitioned by ingest date for reproducible backfills.
\end{itemize}

\textbf{Hot path (partially implemented):}
\begin{itemize}[leftmargin=1.5em]
  \item stream branch represented by \texttt{stream/ids\_alerts/} ingestion,
  \item current source is synthetic IDS events placeholder while keeping the path ready for real sensor/replay input.
\end{itemize}

\textbf{Warm path (planned scaffold present):}
\begin{itemize}[leftmargin=1.5em]
  \item \texttt{warm/stream\_aggregates/} prefix exists in landing setup,
  \item aggregation/materialization logic is reserved for next iteration (trusted/warm transformations).
\end{itemize}

\section{Implemented Technologies and Components}
\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Orchestration}: Apache Airflow DAGs for scheduled ingestion.
  \item \textbf{Object storage}: MinIO as landing zone (\texttt{s3://landing}).
  \item \textbf{Messaging infra available}: Kafka + Zookeeper (currently prepared, stream placeholder active).
  \item \textbf{Metadata/state}: Postgres for Airflow metadata; JSON sidecars + manifests for data lineage.
  \item \textbf{Runtime}: Docker Compose stack for local reproducible execution.
  \item \textbf{Ingestion code}: Python source-specific adapters under \texttt{ingestion/batch/} and dataset importer under \texttt{ingestion/imports/}.
\end{itemize}
